# Тестирование разделено на 2 модуля

- Валидация и синхронизация схемы и аргументов вызова
- Обращение через агрегатор к выбранной модели (OpenRouter, KIA)

## Валидация происходит в несколько этапов

- Проверка сигнатуры функции на правильность: берем имя и словарь аргументов
- Разбираем функцию с помощью модуля `ast` для составления карты первичных вызовов аргументов, пометка `required` аргументов
- Как идея: добавить проверку импортов на нежелательные элементы (из документации)

## Проверка JSON схемы происходит следующим образом

- Проверка наличия правильности ключей
- Проверка обязательных ключей (`default`, `debug`, ...)
- Проверка на наличие неподдерживаемых ключей (`oneOf`, `allOf`, ...)
- Составление карты: имя аргумента — тип, начальное значение

## Синхронизация схемы и функции

- Проверка на наличие обязательных вызовов `required`
- Проверка на наличие начальных значений в вызовах: `arguments.get("debug", False)`
- Проверка на наличие избыточных ключей (не объявленных в функции)
- Проверка на отсутствие необязательных ключей (`Warning`): работа может продолжиться

## Цель и применение

- Применение на этапе `push` в основную ветку разработки (`dev`) при `pull request` -> `branch main`
- Цель: перенести скриптовые действия от телеграм бота в GitHub. Разработчик сам будет видеть мелкие нестыковки, освобождая ревьюера от простой рутинной работы
- При проверке в боте, считаю, чем меньше условий проверки, тем она будет качественнее
